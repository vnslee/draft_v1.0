# 오토금융 진출 진단 대시보드 — 설계 대화 기록

> 산문 규제 비교에서 출발해, 유사도 기반 진출 의사결정 엔진과 프로토타입 대시보드까지 발전시킨 설계 논의 정리본.

## Q1. 산문으로 된 글의 키워드를 찾아 나라끼리 비교하려는데, 쓸 만한 라이브러리가 있나?

**핵심 접근법: "키워드 추출 → 구조화 → 비교"**

산문 텍스트를 국가 간 비교하는 작업은 3개 레이어로 나뉜다.

**① 키워드/개체 추출 (NLP)**

| 라이브러리 | 언어 | 특징 |
|---|---|---|
| spaCy | Python | 실무 표준, 다국어 모델 지원 |
| KeyBERT | Python | BERT 기반 키워드 추출, 품질 높음 |
| YAKE | Python | 비지도학습, 언어 무관 |
| KoNLPy | Python | 한국어 특화 |

**② 의미 기반 유사도 비교**
- `sentence-transformers` — 문장을 벡터화해 국가별 정책 문서를 수치로 비교
- OpenAI Embeddings / Claude API — 산문을 구조화된 JSON으로 변환하는 데 강력

**③ 비교 구조화**
- Pandas — 추출된 키워드를 테이블로 정리
- LangChain — 여러 문서를 파이프라인으로 처리

**추천 조합**
```
산문 문서 (각국 법령/규정)
    ↓
Claude/GPT API로 구조화 (JSON 변환)
    ↓
Pandas로 국가별 비교 테이블
    ↓
시각화 (Excel or 대시보드)
```

> 다루려는 문서 성격(법령/규정, 시장 리포트, 라이선스 요건)에 따라 접근법이 달라진다.

## Q2. 규제·라이선스·인허가 요건 등 전방위로 비교하고 싶다

**현실적 난관 먼저**
- ❌ 각국 문서가 "같은 항목"을 다른 용어로 표현 (예: 최저자본금 = Minimum Capital = Modal Minimum = 注册资本)
- ❌ 문서 구조가 나라마다 완전히 다름
- ❌ 일부 국가는 영어 공식 문서 자체가 없음

**권장 아키텍처**
```
[각국 원문 수집]
    ↓
[Claude API] ← 핵심 엔진
 "이 문서에서 자본금/라이선스종류/심사기간/갱신주기/규제기관 추출"
    ↓
[표준 스키마 JSON 변환]  {country: "VN", min_capital: "...", license_type: [...]}
    ↓
[Pandas DataFrame 통합]
    ↓
[비교 테이블 / 시각화]
```

**표준 스키마 비교 항목 후보 (캐피탈 해외진출 기준)**

| 카테고리 | 세부 항목 |
|---|---|
| 인허가 | 라이선스 종류, 소관 규제기관, 심사기간 |
| 자본 요건 | 최저자본금, 외국인 지분 한도 |
| 운영 제한 | 취급 가능 상품, 금리 상한, 담보 규정 |
| 보고 의무 | 보고 주기, 제출 기관, 언어 요건 |
| 갱신/유지 | 갱신 주기, 유지 조건, 위반 제재 |

## Q3. 권역별로만 비교한다 (유럽↔유럽, 미주↔미주, 아태↔아태)

권역별 비교면 설계가 단순해진다. 단, 권역 내에서도 **레이어**를 나눠야 비교가 의미 있다.

```
유럽 → EU 공통 규제(CRD, MiFID) + 각국 로컬 규제 혼재
미주 → 연방법 + 주법 이중 구조 (특히 미국)
아태 → 국가별 편차 최대 (싱가포르 vs 미얀마)
```

**권역별 비교 시스템 구조**
```
권역 선택 (유럽 / 미주 / 아태)
    ↓
공통 상위 규제 레이어 추출 (EU Directive, 연방법 등)
    ↓
국가별 로컬 규제 추출
    ↓
표준 스키마로 정규화
    ↓
권역 내 국가 간 비교 테이블
```

## Q4. 대시보드 내에 로직을 내재화하고 싶다

```
[문서 입력 레이어]  각국 규제 문서 (PDF/URL/텍스트)
    ↓
[추출 로직 레이어] ← Claude API  키워드 추출 + 표준 스키마 변환
    ↓
[저장 레이어]  구조화 DB (항목별 정규화 데이터)
    ↓
[대시보드 레이어]  권역 선택 → 국가 비교 테이블/차트
```

**짚어야 할 핵심: 업데이트 주기**
- ❶ 규제 문서가 바뀔 때마다 → 재추출 자동화 필요
- ❷ 분기/반기 수동 업데이트 → 반자동으로 충분
- ❸ 고정 스냅샷 비교 → 정적 데이터로 가능

**플랫폼별 로직 내재화 적합도**

| 플랫폼 | 로직 내재화 | 적합도 |
|---|---|---|
| React 커스텀 | 완전 자유도 | ★★★ |
| Streamlit | Python 로직 직결 | ★★★ |
| Power BI | 제한적 | ★★ |
| Tableau | 매우 제한적 | ★ |

## Q5. 진출국은 진출 준비 시점 규제를 저장, 미진출국은 비교 당시 데이터를 조사하는 시나리오다

```
진출 국가                    미진출 국가
─────────────────────        ─────────────────────
진출 준비 시점 규제           비교 시점 현행 규제
(스냅샷 저장)                (실시간/최신 조사)
       └──────── 비교 ──────────────┘
```

**반드시 짚어야 할 리스크 3가지**
- ❶ **시점 불일치** — 진출국은 과거 스냅샷, 미진출국은 현재. "공정한 비교"가 아닐 수 있고 의사결정자가 이 차이를 인지해야 한다.
- ❷ **진출국 규제 변경 추적 누락** — 스냅샷이 고정되면 "현재 우리가 맞는 규제"인지 알 수 없다.
- ❸ **비교 기준선이 흔들림** — 시점이 다르면 대시보드 보는 사람이 오해할 여지.

**권장 데이터 모델**
```
국가 테이블
├── country_id
├── region (유럽/미주/아태)
├── status (진출/미진출)
└── regulations[]
     ├── snapshot_date      ← 반드시 필요
     ├── data_type          ← "진출준비시점" / "현행"
     ├── items{}            ← 실제 규제 내용
     └── is_current         ← 최신 여부 플래그
```

> 진출국도 **진출 시점 스냅샷**(의사결정 당시 근거 보존)과 **현행 규제 트래킹**(운영 중 변경 모니터링) 두 레이어를 함께 가져가는 게 맞다.

## Q6. 진출 시점 데이터로 비교하려는 이유는 그 당시 비용/기간과 비교해 신규 진출 공수를 추정하기 위함이다

→ 단순 규제 비교가 아니라 **진출 비용/기간 추정 모델**. 컨설팅에서 **Benchmark Model**이라 부르는 구조.

```
진출 시점 규제 + 실제 소요 비용/기간 (실적)
    ↓
"규제 복잡도 → 비용/기간" 상관관계 도출
    ↓
미진출국 현행 규제 입력
    ↓
신규 진출 예상 비용/기간 추정
```

**진출 실적 데이터에서 자주 누락되는 항목**
- ❌ 공식 비용만 기록 (비공식 비용 누락)
- ❌ 인허가 기간만 기록 (사전 준비 기간 누락)
- ❌ 초기 비용만 기록 (1~2년차 운영 고정비 누락)
- ❌ 본사 지원 인력 공수 미반영

> 캐피탈 업계 특성상 현지 로펌/컨설팅 비용이 전체의 30~40%를 차지하기도 한다.

## Q8. 국가별 조사 데이터 — 큰 카테고리를 실무 수준으로 쪼개 정리

> 5개 카테고리는 성격이 다르다. 1·3은 시장 데이터(수치 중심), 2는 규제(정성+정량), 4는 경쟁사, 5는 사용자 입력값.
> 정량은 직접 비교 가능하지만, 정성(규제)은 표준 스키마 정규화가 필요하다.

딜러 채널 구조는 논의 끝에 **1-2로 편입**하기로 했다. (공식 통계가 약하고 처음 구축 시 카테고리를 잘게 쪼개면 데이터 채우기가 부담되므로, 우선 시장조사 하위로 묶고 데이터가 쌓이면 독립시키는 방향.)

## Q10. 에이전트 대시보드 전체 설계 (목적 정의)

> 단순 비교 대시보드가 아니라 **유사도 기반 진출 의사결정 엔진**.

**전체 아키텍처**
```
[기준 데이터 레이어]  진출국 실적 DB
├── 진출 준비기간/비용 (실적)
├── 시스템 정보 (인프라/상품/코어)
└── 진출 시점 규제 스냅샷
        ▼
[조사 레이어] ← 에이전트가 실시간 수집
대상국 정보 (시장/규제/환경/시스템) ※ 공식 출처만
        ▼
[추출·정규화 레이어]  산문 → 값 변환
        ▼
[유사도 엔진] ← 핵심  카테고리별 스코어링 + 가중치
├─ 임계점 통과 → 이식/확장 시나리오 + 비용 추정
└─ 임계점 미달 → 경쟁사/로컬사 심층조사
        ▼
[리포트 레이어]  근거 기반 보고서 (요약+AI인사이트+상세+출처+조사일자+가중치)
```

**기능 1 — 단일 국가 진출 진단**
```
INPUT: 대상국
STEP 1. 권역 매칭 — 진출국 중 같은 권역 검색
STEP 2. 카테고리별 유사도 산출 (시장/규제/환경/시스템) × 가중치 → 종합 점수
   ├── 점수 ≥ 임계값(예: 70) → "이식+확장 가능" + 비용 = 비교국 실적 × 유사도 비율
   └── 점수 < 임계값 → 경쟁사/로컬사 심층조사 (업력·딜러망·직원수·솔루션)
```

**기능 2 — 권역 순위**: 기능 1을 권역 내 전 국가에 루프 → 종합 점수 정렬. 순위표에는 "왜 이 순위인지" 카테고리별 분해를 함께 표시.

### 반드시 짚어야 할 설계 리스크 5가지
- ❶ **유사도 ≠ 비용 비율** — "유사도 70%니까 비용 70%"는 위험. 시스템 이식은 유사도가 높을수록 비용이 줄고 **비선형**(90→95%는 확 줄지만 70→75%는 별 차이 없음). 유사도→비용 변환 함수를 별도로 둔다.
- ❷ **이식 가능은 카테고리마다 별도 판정** — 종합 70점이어도 시스템 유사도 40점이면 코어 이식 불가. **시스템 카테고리는 별도 게이트(gate)**로 둔다.
- ❸ **규제는 유사도가 아니라 충족/불충족** — 외국인 지분 100% vs 49%는 "유사한 정도"가 아니라 사업이 되냐 안 되냐. 일부 규제 항목은 **킬 스위치**(통과 못 하면 진출 불가)로 뺀다.
- ❹ **신뢰도가 점수에 반영** — 공식 출처와 추정이 같은 가중치를 가지면 안 된다. 신뢰도 등급을 **점수 보정 계수**로 넣는다.
- ❺ **"공식 출처만" 요건이 가장 큰 난관** — 뉴스·블로그 배제, 법령·재무제표·감독기관만. 출처 도메인을 **화이트리스트로 필터링**하고 못 찾으면 비운다(억지로 채우지 않음).

### 데이터 모델 (핵심 테이블)
```
COUNTRY          country_id, region, entry_status
ENTRY_RECORD     prep_period, prep_cost, entry_date, system_info, reg_snapshot_date  ※진출국만
RESEARCH_DATA    country_id, category, item, value, source_url, source_type,
                 survey_date, confidence_grade, is_official
WEIGHT_RULESET   ruleset_id, category, item, weight, is_default
SIMILARITY_RESULT target_country, compared_country, category_scores{}, total_score,
                 verdict, cost_estimate, ruleset_id(스냅샷), survey_date, report_id
REPORT           summary, ai_insight, detail[], sources[], weight_snapshot, generated_date
```

> 핵심은 SIMILARITY_RESULT와 REPORT에 ruleset과 survey_date를 **스냅샷으로 박는 것**. 가중치를 바꿔도 과거 보고서는 그때 기준으로 재현된다.

### 스코어링 공식
```
카테고리 점수 = Σ(항목 유사도 × 항목 가중치 × 신뢰도계수)
종합 점수    = Σ(카테고리 점수 × 카테고리 가중치)
              단, 시스템 게이트 / 규제 킬스위치 선통과 조건
진출비용 추정 = 비교국 실적비용 × f(시스템 유사도)   ※ f는 비선형 변환함수
```

## Q11. 디폴트 셋 + 설정 화면, 산출물은 별도 파일로

**확정 사항**
- 임계값·가중치: 디폴트 셋으로 잡되 대시보드에 설정 화면 제공
- 진출국 실적: 현재 4개국(권역별 1개국, 아태 2개국) → 당장은 "유사도 몇 % → 기존 대비 몇 % 세이브" 구간표로, 점진 고도화
- 산출물: **설계 문서 / 데이터 카테고리 정의서 / 데이터 스키마**를 각각 다른 파일로

**생성된 3개 파일**
- **01 설계문서** — 아키텍처, 기능1/2 명세, 스코어링 로직, 디폴트 셋, 설계 리스크와 대응, 화면 구성, 고도화 로드맵
- **02 데이터카테고리정의서** — 시장/규제/환경/시스템 4개 카테고리 항목 + 유사도 유형(연속/범주/이진/참고)
- **03 데이터스키마** — 가중치 룰셋 버전 관리, 조사일자 추적, 공식 출처 강제, 보고서 가중치 스냅샷

> 가중치 재현성을 **"참조 + 사본" 이중**으로 설계: 사용자가 가중치를 바꾸면 기존 룰셋을 수정하지 않고 새 버전을 만들고, 보고서에는 룰셋 ID 참조와 전체 사본을 둘 다 박는다.

**이후 정의서 보강 (점진적)**
- 경쟁사 금리 현황 → **Range(최저~최고 %)**, 참고용. 스키마 `data_type`에 `RANGE` 추가, 값은 `{min, max}`로 저장.
- 오토금융 성장률(CAGR) → 참고용
- 고객 세그먼트 → 1-1 시장 일반에 추가. **개인:법인 비율은 연속(점수 반영)**, 기타 특성은 참고용 다중값.

## Q12. 산문으로 된 규제를 어떻게 비교하나?

> 결론: **산문을 직접 비교하지 않는다.** 산문을 구조화된 값으로 바꾸고, 그 값을 비교한다.

```
원문 산문 (각국 법령)
  ↓ ① 추출 (Extraction)        값 + 근거 + 조건/예외 + 출처를 JSON으로 (추측 금지, 못 찾으면 null)
  ↓ ② 정규화 (Normalization)   통화·단위·코드 통일 (EUR/PLN → USD, "갱신 5년" → 주기_년=5)
  ↓ ③ 비교 (Comparison)        유형별로 방법이 다름
```

**유형별 비교 방법**

| 유사도 유형 | 비교 방법 | 예시 |
|---|---|---|
| 연속 | 수치 거리 → 유사도 점수 | 인허가 90일 vs 120일 → 75% 유사 |
| 범주 | 같으면 일치, 다르면 불일치 | GDPR vs GDPR → 100% |
| 이진 | 충족/불충족 (킬스위치) | 지분 100% 허용? → Y/N |
| 참고 | 비교 안 함, 정보 표시만 | 경쟁사 금리 Range |

**산문에만 있는 "뉘앙스" 보완**
- 의미 임베딩(semantic embedding) 보조 점수 — 규제 조항 전체를 벡터화해 "전반적으로 얼마나 유사한 규제 환경인가"를 연속 보조점수로 산출 (숫자 추출과 별개 레이어)
- 조건·예외를 별도 필드로 추출 (`value: 49, condition: 현지 JV 시 100% 가능`) 후 사람이 검토하도록 보고서에 플래그

> 추출·정규화는 LLM에 맡기되, 조건부 규정과 킬스위치 항목은 사람이 확인하는 **반자동(human-in-the-loop)**이 현실적. 규제 오판은 진출 실패로 직결된다. → 이 내용은 설계 문서 **5장**으로 추가됨.

## Q13. 공식 출처 화이트리스트 — 너무 엄격하면 데이터가 안 모이지 않나?

> 화이트리스트의 진짜 딜레마: 너무 엄격하면 데이터가 안 모이고, 너무 느슨하면 공식 출처 원칙이 무너진다. **이분법이 아니라 출처 등급제**로 푼다.

**4단계 등급**

| 등급 | 출처 | 처리 |
|---|---|---|
| Tier 1 | 공식 1차 (법령·재무제표·감독기관) | 그대로 사용 |
| Tier 2 | 준공식 (협회·국제기구·빅4) | 사용하되 "공식 미확보" 플래그 + 신뢰도 계수로 자동 감점 |
| Tier 3 | 참고 | 감점 폭 확대 |
| 차단 | 뉴스·블로그 | 코드에서 원천 배제 |

**두 가지 원칙**
- 화이트리스트는 **프롬프트가 아니라 코드 레벨에서 강제** (LLM이 출처를 공식이라 우기거나 URL을 지어내는 걸 막기 위해)
- 공식 1차 출처를 못 찾아도 항목을 비우지 않고 Tier 2·3로 채우되 감점 → 신흥국·비영어권에서 1차 법령DB가 없어도 시스템이 멈추지 않음

> 화이트리스트의 목적은 "데이터를 막는 문지기"가 아니라 **"출처 약한 데이터에 자동으로 할인을 매기는 채점관"**.
> 스키마(03번)도 정합 조정: `RESEARCH_ITEM`에 `source_tier`·`official_gap_flag` 추가, 기존 "공식 미발견 시 비워둔다" 규칙을 "Tier 2·3로 채우되 감점"으로 수정.

| 파일 | 역할 | 대상 |
|---|---|---|
| 01 설계문서 | "왜/무엇을" 만드는가 | PM·아키텍트·전사 공유 |
| 02 데이터카테고리정의서 | "무엇을 담는가" | 조사 담당·도메인 전문가 |
| 03 데이터스키마 | "어떻게 저장하는가" | 백엔드·DB 개발자 |
| 04 화면설계 흐름 | 화면 간 이동 구조 | 기획·프론트 |
| 프로토타입 (HTML) | 목업 데모 | 전체 |

**개발 순서 (6단계)**
1. 합의 (01) — 디폴트 가중치·임계값·게이트/킬스위치 정책 동의
2. 데이터 모델 구축 (03) — 테이블 DDL 변환, WEIGHT_RULESET 버전 관리부터, 02 항목을 CATALOG_ITEM 시드로 적재
3. 조사·추출 파이프라인 (01-5장 + 02) — LLM 추출 프롬프트에 02 항목명·유사도 유형·타입 사용
4. 스코어링 엔진 (01-6장) — **킬스위치 선검증 → 게이트 → 카테고리 → 종합** 순서
5. 기능 구현 (01-4장)
6. 보고서·화면 (01-4.3 + 9장)

> 세 문서는 **항목 ID로 연결**된다. 02 "외국인 지분 한도" → CATALOG_ITEM `REG_FOREIGN_OWNERSHIP` → RESEARCH_ITEM 참조 → 스코어링이 이 ID로 가중치 조회. STEP 2에서 모든 항목에 고유 ID를 먼저 부여하는 게 전체의 뼈대.

**처음 구축 시 함정 3가지**
1. 스코어링 엔진부터 만들고 싶은 유혹 — 데이터 모델·추출 파이프라인이 없으면 넣을 데이터가 없다. 순서를 지킬 것.
2. 가중치 버전 관리를 "나중에"로 미루기 — 상수로 박으면 설정 화면 붙일 때 보고서 재현성이 깨진다.
3. 공식 출처 필터를 LLM에 맡기기 — 프롬프트로는 부족, **도메인 화이트리스트를 코드 레벨에서 강제**.
