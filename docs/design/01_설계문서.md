# 오토금융 해외진출 에이전트 대시보드 — 설계 문서

> 버전 v0.1 (초안) · 작성일 2026-06-14 · 분류: 내부 설계 문서

---

## 1. 목적과 범위

### 1.1 한 줄 정의
진출국의 실적·시스템 데이터를 기준점으로 삼아, **미진출 국가의 진출 난이도·비용을 유사도 기반으로 추정**하고, 권역 내 국가들의 **진출 우선순위를 줄세우는** 의사결정 지원 에이전트.

### 1.2 핵심 가치
- 신규 진출 시 "기존 시스템을 이식·확장할 수 있는가"를 정량 판정
- 이식 가능 시 → 기존 진출 비용 대비 절감 비율로 신규 비용 추정
- 이식 불가 시 → 현지 경쟁사/로컬사 심층조사로 자동 전환
- 모든 결과는 **공식 출처 기반 근거 보고서**로 산출

### 1.3 범위 밖 (현 단계 제외)
- 실제 인허가 대행, 법률 자문
- 실시간 재무 시뮬레이션(수익성 모델링)은 후속 고도화 대상
- 뉴스·블로그 등 비공식 출처 기반 조사

## 2. 데이터 자산 (전제)

본 에이전트는 두 종류의 기준 데이터를 전제로 동작한다.

### 2.1 진출국 실적 데이터 (Entry Record)
이미 진출한 국가에 대해 보유:
- 진출 준비 기간 (사전조사 ~ 인허가 완료)
- 진출 비용 (라이선스, 법인설립, 시스템 구축, 현지 자문 등)
- 시스템 정보 (인프라, 취급 상품, 코어 시스템 구성)
- 진출 시점의 규제 스냅샷

> 진출 권역 (북미, 남미, 유럽, APEC, 중동), 실적 데이터 보유 국가 (APEC-호주, 북미-미국, 유럽-영국)

### 2.2 조사 데이터 (Research Data)
대상국에 대해 에이전트가 병렬로 수집:
- 시장 / 규제 / 환경 / 시스템 4개 카테고리
- 각 agent가 각 카테고리별로 조사 진행 후 결과 데이터 생성
- 각 항목에 **출처, 출처유형, 조사일자, 신뢰도등급** 부착
- 공식 출처(법령·재무제표·감독기관)만 채택
- 조사 결과 데이터는 항상 동일한 포맷으로 유지되어야함.
- "02_데이터카테고리정의서.md"와 "03_데이터스키마정의서.md" 참조하여 일관된 결과물 생성 필요
- 최종 결과는 DB에 저장

## 3. 전체 아키텍처

```
[기준 데이터 레이어]
  진출국 실적 DB + 시스템 정보 + 규제 스냅샷
        │
        ▼
[조사 레이어]  ← 에이전트 실시간 수집 (공식 출처 필터)
  대상국 시장/규제/환경/시스템 데이터 (상당수 산문)
        │
        ▼
[추출·정규화 레이어]  ← 산문 → 구조화 값 (5장)
  LLM 추출(값+근거+조건) → 단위·통화·코드 정규화
        │
        ▼
[유사도 엔진]  ← 핵심
  카테고리별 유사도 산출 × 가중치 × 신뢰도계수
  + 시스템 게이트 + 규제 킬스위치 + 임베딩 보조점수
        │
        ├─ 임계 통과 → 이식·확장 시나리오 + 비용 추정
        └─ 임계 미달 → 경쟁사/로컬사 심층조사
        │
        ▼
[리포트 레이어]
  요약 + AI인사이트 + 상세 + 출처 + 조사일자 + 가중치 스냅샷
```

## 4. 기능 명세

### 4.1 기능 1 — 단일 국가 진출 진단

| 단계 | 처리 내용 |
|---|---|
| INPUT | 대상국 (예: 체코) |
| STEP 1 | 권역 매칭 — 대상국 권역 식별 후 동일 권역 진출국 탐색 |
| STEP 2 | 카테고리별 유사도 산출 (시장/규제/환경/시스템) |
| STEP 3 | 가중치·신뢰도 적용 → 종합 유사도 점수 |
| STEP 4-A | 점수 ≥ 임계값 & 게이트 통과 → "이식·확장 가능" 판정 + 비용 추정 |
| STEP 4-B | 점수 < 임계값 또는 게이트 실패 → 경쟁사/로컬사 심층조사 |
| OUTPUT | 근거 보고서 + 결과 레코드 저장 |

**비교 대상이 여러 개일 때**: 동일 권역에 진출국이 2개 이상이면(예: 아태), 각각과 유사도를 산출하고 **가장 높은 비교국**을 기준으로 판정한다. 보고서에는 전체 비교 결과를 함께 기록한다.

### 4.2 기능 2 — 권역 진출 우선순위

| 단계 | 처리 내용 |
|---|---|
| INPUT | 권역 (미주 / 아태 / 유럽) |
| STEP 1 | 권역 내 미진출국 목록 확보 |
| STEP 2 | 각 국가에 대해 기능 1 수행 |
| STEP 3 | 종합 점수 기준 정렬 → 순위 부여 |
| OUTPUT | 순위표 + 카테고리별 점수 분해 + 권역 종합 보고서 |

> 순위표에는 반드시 **카테고리별 점수 분해**를 포함한다. "왜 이 순위인지"(시스템 때문인지 시장 때문인지)가 보여야 의사결정 신뢰도가 확보된다.

### 4.3 서브 기능

**(A) 근거 보고서 생성**
- 구성: 요약(AI 인사이트 포함) + 상세(항목별 자료) + 출처 + 조사일자 + 가중치 스냅샷
- 출처 원칙: **국가/기관 공식 자료만** (재무제표, 법령, 감독기관 공시). 뉴스·블로그 불가
- 공식 출처 미발견 항목은 억지로 채우지 않고 "데이터 없음"으로 표기

**(B) 가중치 설정**
- 카테고리·항목별 가중치는 **디폴트 셋** 제공
- 사용자가 설정 화면에서 변경 가능 → 변경 시 점수 재계산
- 보고서 생성 시점의 가중치를 **스냅샷으로 저장** (재현성 보장)

**(C) 조사일자 관리**
- 모든 조사 데이터에 조사일자 부착
- 진출국 스냅샷(과거)과 대상국 현행 데이터의 **시점 격차를 보고서에 명시**

## 5. 규제·산문 데이터 처리 (Extraction & Normalization)

> 규제는 대부분 법령 산문 형태다. 산문끼리 직접 "유사도"를 계산하지 않는다. **산문 → 구조화 값 → 비교**의 3단계로 처리한다. 6장 스코어링은 이 단계를 거친 값을 입력으로 받는다.

### 5.1 원칙: 비교는 산문이 아니라 정규화된 값끼리

```
법령 산문
  → ① 추출(Extraction)    : 항목별 값 + 근거 + 조건/예외
  → ※ 출처 검증(코드 레벨)  : 도메인 화이트리스트 등급 판정
  → ② 정규화(Normalization): 단위·통화·코드 통일
  → ③ 비교(Comparison)     : 유형별(연속/범주/이진) 유사도 산출
  → + 의미 임베딩 보조점수  : 조항 전체의 의미 유사도
  → 카테고리 점수 + 사람 검토 플래그
```

### 5.2 ① 추출 (LLM 기반)

법령 원문에서 정의서(02번)의 항목을 LLM이 JSON으로 추출한다.

- **값(value)**, **근거 문장(evidence)**, **출처(source)**를 함께 반환
- 못 찾으면 `null`, **추측 금지** (공식 출처 원칙)
- 조건·예외는 별도 필드(`condition`)로 분리

```
출력 예시
{
  "외국인지분한도": {
    "value": 49,
    "condition": "현지 금융기관과 JV 시 감독당국 승인하 100% 가능",
    "evidence": "외국인투자법 제12조 ...",
    "source": "법령명/조문/URL",
    "confidence": "OFFICIAL"
  },
  "최저자본금": { "value": null, "evidence": null }
}
```

근거 문장과 출처를 함께 뽑는 것이 근거 보고서·공식 출처 요건을 충족한다.

### 5.3 출처 검증 — 화이트리스트는 코드로 강제하되, 등급제로 운용

**원칙: 출처 검증을 LLM 프롬프트에 맡기지 않는다.** "공식 출처만 써"라는 지시만으로는 LLM이 비공식 출처를 공식이라 우기거나 URL을 지어낼 수 있다. 출처 판정은 **코드 레벨에서 도메인을 검증**해 강제한다(설계 리스크 5번).

```
LLM이 source URL/문서명 반환
        ↓
[코드 레벨 출처 검증기]  ← 프롬프트 아님
  도메인 화이트리스트 대조 → 출처 등급 판정
        ↓
등급에 따라 신뢰도 계수 부여 (6.3 연동)
```

**딜레마: 화이트리스트만 고집하면 데이터가 안 모인다.** 공식 1차 출처(법령DB·감독기관·재무제표)에만 의존하면, 신흥국이나 비영어권은 데이터가 통째로 비어버린다. 그렇다고 아무 출처나 받으면 공식 출처 원칙이 무너진다. 그래서 **이분법이 아니라 등급제(tiered allowlist)**로 운용한다.

| 등급 | 출처 예시 | 처리 | 신뢰도 계수 |
|---|---|---|---|
| **Tier 1 (공식 1차)** | 법령DB, 감독기관 공시, 관보, 재무제표 원본 | 그대로 채택 | 1.0 |
| **Tier 2 (준공식)** | 산업협회, 국제기구(OECD·IMF), 공인 통계기관, 빅4 보고서 | 채택하되 표시 | 0.8 |
| **Tier 3 (참고)** | 1차 출처를 인용한 2차 자료 | 1차 출처 역추적 시도 → 실패 시 플래그 | 0.5 |
| **차단** | 뉴스·블로그·위키·익명 | **코드에서 배제** | 0 (점수 제외) |

**운용 규칙**:
- 차단 등급(뉴스·블로그 등)은 코드에서 원천 배제. 프롬프트 신뢰 금지.
- Tier 2·3로 채운 항목은 **"공식 출처 미확보" 플래그**를 달아 보고서에 표시한다. 데이터는 확보하되 "이건 1차 출처가 아니다"를 의사결정자가 알게 한다.
- Tier 1을 못 찾았다고 항목을 비우지 않는다. **Tier 2·3로 채우고 신뢰도 계수로 자동 감점**한다. 이렇게 하면 데이터 공백을 메우면서도 점수에는 출처 약함이 반영된다.
- 화이트리스트는 국가별로 다르다(나라마다 감독기관·법령DB 도메인이 다름). 국가별 도메인 목록을 **설정으로 관리**하고 점진 확장한다(로드맵 4번).

> **핵심**: 화이트리스트의 목적은 "데이터를 막는 것"이 아니라 "출처 약한 데이터를 점수에서 자동 할인하는 것"이다. 등급제 + 신뢰도 계수 연동이 데이터 공백과 출처 신뢰도 사이의 균형을 맞춘다.

### 5.4 ② 정규화

추출값은 나라마다 단위·기준이 달라 그대로 비교 불가. 표준 형태로 변환한다.

| 변환 유형 | 예시 |
|---|---|
| 통화 통일 | EUR 730만 / PLN 4천만 → USD 환산 |
| 단위 통일 | "5년마다 갱신" / "갱신 의무 없음" → 갱신주기(년): 5 / 0 |
| 코드값 매핑 | 라이선스 종류 텍스트 → 표준 코드 |
| 기준일 정렬 | 환율·물가 기준일 명시 |

> 정규화 결과는 스키마의 `value_normalized` 필드에 저장(03번 문서 4장). RANGE 타입은 {min, max}.

### 5.5 ③ 비교 (유형별)

정의서에서 항목마다 지정한 유사도 유형에 따라 비교 방식이 갈린다.

| 유사도 유형 | 비교 방법 | 예시 |
|---|---|---|
| 연속 | 수치 거리 → 유사도 점수 | 인허가 90일 vs 120일 → 75% |
| 범주 | 일치/불일치 | GDPR vs GDPR → 100% |
| 이진 | 충족/불충족(킬스위치) | 외국인 100% 가능? → Y/N |
| 참고 | 비교 안 함, 보고서 표시만 | 경쟁사 금리 Range |

### 5.6 산문 뉘앙스 처리 — 조건부·예외 규정

규제는 숫자로 떨어지지 않는 조건부 규정이 많다(예: "원칙 49%, JV 시 100% 가능"). 두 가지로 보완한다.

1. **의미 임베딩 보조점수**: 규제 조항 전체를 벡터화해 "전반적 규제환경 유사도"를 연속 보조점수로 산출. 숫자 추출과 별개 레이어.
2. **조건/예외 별도 필드 + 검토 플래그**: 추출 시 조건을 분리 저장하고, 사람이 확인하도록 보고서에 플래그를 단다.

### 5.7 반자동(Human-in-the-Loop) 원칙

완전 자동화는 규제 오판 위험, 완전 수동은 비효율. 추출·정규화는 LLM이 처리하되, **조건부 규정과 킬스위치 항목은 사람이 최종 확인**한다. 규제 오판은 진출 실패로 직결되므로 이 항목들은 자동 판정에 맡기지 않는다.

## 6. 스코어링 로직

### 6.1 기본 공식

```
항목 유사도(0~100) × 항목 가중치 × 신뢰도 계수
        ↓ 카테고리 내 합산
카테고리 점수
        ↓ 카테고리 가중치 적용 후 합산
종합 점수(0~100)
```

### 6.2 게이트 & 킬스위치 (단순 평균의 함정 방지)

| 메커니즘 | 설명 | 예시 |
|---|---|---|
| **시스템 게이트** | 시스템 카테고리 점수가 하한 미만이면 "이식 가능" 판정 불가 | 종합 75점이어도 시스템 40점이면 코어 이식 불가 |
| **규제 킬스위치** | 특정 규제 항목이 불충족이면 진출 자체 불가 처리 | 외국인 지분 한도가 사업모델을 막는 경우 |

> **설계 원칙**: 시장·환경은 "유사한 정도"(연속값)로 보지만, 규제는 일부 항목이 "되냐/안 되냐"(이진값)다. 규제 카테고리는 유사도 점수와 별개로 킬스위치 항목을 둔다.

### 6.3 신뢰도 계수
공식 출처 확인 항목과 추정 항목이 같은 무게를 갖지 않도록 보정한다.

| 신뢰도 등급 | 계수(예시) |
|---|---|
| 공식(법령/재무제표/감독기관) | 1.0 |
| 준공식(협회/공인 통계) | 0.8 |
| 추정/인터뷰 | 0.5 |

### 6.4 유사도 → 비용 변환

현재 단계는 **구간별 비율표**로 시작하고, 진출국 실적이 쌓이면 함수로 고도화한다.

| 종합 유사도 | 신규 진출 비용(비교국 실적 대비) | 의미 |
|---|---|---|
| 90% 이상 | 30~40% | 거의 그대로 이식 |
| 80~90% | 40~55% | 소폭 확장 |
| 70~80% | 55~70% | 부분 재구축 |
| 70% 미만 | — (이식 부적합, 심층조사 전환) | 신규 구축 검토 |

> ⚠️ **주의**: 유사도와 비용은 선형 비례가 아니다. 90→95% 구간의 비용 절감이 70→75% 구간보다 크다(비선형). 위 표는 출발점이며, 실적 데이터로 지속 보정한다.
>
> ⚠️ **표본 한계**: 현재 진출국 4개국 기준이라 통계적 함수화는 무리. 구간 비율표로 운용하며 케이스를 누적한다.

## 10. 후속 고도화 로드맵

1. **변환 함수화**: 진출 케이스 누적 시 구간표 → 회귀 함수
2. **수익성 모델 연동**: 비용 추정 + 시장규모/금리/LTV → ROI 시뮬레이션
3. **규제 변경 트래킹**: 진출국 현행 규제 모니터링(스냅샷 vs 현행 비교)
4. **자동 출처 검증**: 공식 도메인 화이트리스트 자동 확장
